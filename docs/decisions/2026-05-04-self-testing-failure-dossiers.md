# Self-Testing Failure Dossiers

**Date:** 2026-05-04
**Status:** active
**Durable:** yes — the run identity, atomic write, and host-asymmetric hook choices are easy to cargo-cult later without the Codex hook spike context.

## Decision

Continue the self-testing loop with runner-written **failure dossiers**. A 0th-managed runner wraps verification commands and, on non-zero exit, writes one immutable dossier at `${VERIFICATION_REPORT_DIR:-verification-report}/runs/<run_id>/dossier.json` using temp-file-then-rename.

The runner owns failure detection. Hooks only surface dossiers; they do not infer Bash failure from `tool_response`. This preserves the hook-spike finding that Codex Bash responses have no reliable exit-code shape.

Each dossier contains `run_id`, `cwd`, `command`, `exit_code`, `started_at`, `finished_at`, truncated stdout/stderr, and optional artifact paths. `run_id` is generated by the caller and must appear in both the tool command and the dossier. Hook adapters only surface a dossier when the current hook payload's `tool_input` names that same `run_id`; the surfaced message includes the current `session_id`, `turn_id`, and `tool_use_id` so stale files cannot masquerade as the current run.

Codex and Claude stay asymmetric at the event layer. Codex uses `PostToolUse` plus `hookSpecificOutput.additionalContext`; Claude uses `PostToolUseFailure` where available. The unified output contract is: after a managed verification command fails, the next agent turn receives a fresh dossier path with the run identity and a short failure summary.

Pre-dispatch tool failures are separate. Hook adapters cannot see rejected tool calls, so `/ship` remains responsible for blocking missing verifier evidence, and the verifier report will gain an explicit `pre_dispatch_tool_failures_reviewed` field before this loop is called complete.

## Constraints

- No single shared mutable dossier file; optional indexes may exist only as derived conveniences.
- Dossier writes are atomic and schema-validated before surfacing.
- Secrets follow the 0th secret-handling contract: output is truncated, never shell-traced, and no secret-resolution commands are introduced.
- User-scope Codex hook configuration is the only validated Codex path until repo-local hook reliability is re-tested.

## Not Doing

- Parsing Codex Bash `tool_response` for failure.
- Auto-installing user-scope hooks from the repo.
- Treating hook coverage as a substitute for `/ship`'s mechanical verifier gate.
