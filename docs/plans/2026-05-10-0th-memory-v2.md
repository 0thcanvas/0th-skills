# 0th Memory v2 Plan

**Decision:** [docs/decisions/2026-05-10-0th-memory-v2.md](../decisions/2026-05-10-0th-memory-v2.md)
**Slices:** 6

## Architecture

- Evidence layer stays in markdown: decisions, KB pages, incidents, failure dossiers, and `CONTEXT.md`.
- Machine layer is generated: brief/profile artifacts plus a local index of memory claims with source paths, timestamps, confidence, and review state.
- Memory capture is event-driven through skill gates, not session-end-driven.
- Repo freshness is a conservative preflight: fetch, fast-forward only when clean, then sync memory against changed sources.
- Exploration freshness uses read-set reconciliation instead of whole-KB comparison.

## Slices

### 1. Memory contract and lifecycle

Define the shared memory vocabulary, lifecycle states, and write gates used by all core skills.
- [x] Memory types cover decisions, observations, root causes, vocabulary, incidents, repo state, and external research.
- [x] Lifecycle states distinguish active, needs review, superseded, archived, and ephemeral memory.
- [x] Skill prompts have a required memory write gate with an explicit `nothing durable` outcome.

### 2. Conservative repo preflight

Every core skill starts from visible repo state and avoids working from stale remote history when a safe fast-forward is available.
- [x] Preflight reports branch, cleanliness, upstream relation, and last fetched state.
- [x] Clean branches behind upstream are fast-forwarded only when the update is safe.
- [x] Dirty or divergent branches produce a visible warning and do not auto-merge, reset, or stash.

### 3. Memory sync after repo updates

Successful upstream updates mark related memory as reviewable without pretending the agent has re-verified changed code.
- [x] Sync records old and new revision identifiers after a fast-forward update.
- [x] Changed sources are mapped to memory claims that cite those paths or symbols.
- [x] Affected claims become `needs_review` until later exploration or tests confirm the new truth.

### 4. Agent brief generation

Agents get a compact project brief as the first read path instead of browsing indexes and raw notes manually.
- [ ] Briefs summarize active decisions, vocabulary, recurring incidents, known root causes, and current repo-state warnings.
- [ ] Brief generation cites source evidence so claims remain traceable.
- [ ] Regeneration is deterministic enough for tests to detect drift in required sections.

### 5. Read-set reconciliation

Code exploration updates memory only where the agent has already spent tokens verifying the relevant area.
- [ ] Explorers record the files, symbols, tests, and claims they inspected.
- [ ] Reconciliation checks only memory tied to that read set.
- [ ] Verified contradictions create a memory update or review note with evidence; unrelated KB content is left untouched.

### 6. Memory eval and backend decision

Choose the storage/search backend only after testing real 0th recall tasks against current and candidate systems.
- [ ] Eval set includes at least 10 real questions about decisions, stale claims, recurring mistakes, repo vocabulary, and changed code behavior.
- [ ] Baselines compare current markdown lookup, a thin 0th-native index, MemPalace-style verbatim retrieval, and agentmemory-style lifecycle/profile behavior.
- [ ] The follow-up decision records whether to keep the thin local layer or adopt an external memory backend.
